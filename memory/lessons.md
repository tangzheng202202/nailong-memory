# Lessons - 经验教训总结

## 🔴 严重（必须避免）

### 1. 配置文件格式不兼容
**时间**: 2026-03-07  
**问题**: 在 `openclaw.json` 中添加自定义 agent 定义导致配置无效  
**原因**: OpenClaw 2026.3.1 版本不支持 `agents.coder` 等自定义 key  
**解决**: 只能使用 `agents.defaults`，自定义 agent 需用 `openclaw agents add` 命令  
**教训**: 修改配置前先验证 schema，不要假设配置格式

### 2. Bindings 配置不生效
**时间**: 2026-03-07  
**问题**: 配置了 5 个飞书群 bindings，但消息都路由到 main Agent  
**原因**: bindings 需要配合 agent 定义，且版本可能不兼容  
**解决**: 改用关键词触发或等待版本更新  
**教训**: 复杂功能先测试再部署，不要假设配置即生效

## 🟡 重要（需要注意）

### 3. Reddit 被限流
**时间**: 2026-03-07  
**问题**: Reddit API 频繁超时，无法获取内容  
**原因**: Reddit 加强反爬，国内 IP 容易被限  
**解决**: 放弃 Reddit，改用国内源（V2EX、B站）  
**教训**: 不要依赖单一数据源，要有备份方案

### 4. 监控频率过高被封
**时间**: 2026-03-07  
**问题**: 每 2 小时监控，部分源请求失败  
**原因**: 请求频率过高触发限流  
**解决**: 降低频率到每 4 小时  
**教训**: 监控频率要合理，宁愿少而精，不要多而滥

### 5. X/YouTube 需要登录
**时间**: 2026-03-07  
**问题**: 无法直接抓取 X/YouTube 热门内容  
**原因**: 需要登录才能看趋势  
**解决**: 改用 RSS 订阅（Nitter、RSSHub）  
**教训**: 抓取前先调研目标站点的反爬策略

## 🟢 建议（优化方向）

### 6. 任务拆分粒度
**时间**: 2026-03-07  
**问题**: 视频网站项目前端任务超时  
**原因**: 任务太复杂，单次执行时间过长  
**解决**: 拆分为更小任务（HTML、CSS、JS 分开）  
**教训**: 子 Agent 任务要小而明确，避免超时

### 7. 记忆写入不规范
**时间**: 2026-03-07 之前  
**问题**: 日志内容杂乱，难以检索  
**原因**: 没有统一的写入规范  
**解决**: 制定 AGENTS.md 记忆写入规范  
**教训**: 从一开始就规范格式，避免后期整理

### 8. 缺少重试机制
**时间**: 持续  
**问题**: 网络波动导致监控失败  
**原因**: 没有失败重试  
**解决**: 添加 3 次重试机制  
**教训**: 网络操作必须有重试，提高稳定性

## ✅ 成功经验

### 1. 国内源优先策略
**时间**: 2026-03-08  
**结果**: V2EX、B站 监控非常稳定  
**经验**: 优先使用无需代理的国内源，可靠性更高

### 2. RSS 替代方案
**时间**: 2026-03-08  
**结果**: 成功获取 X/YouTube 内容  
**经验**: RSS 是稳定的信息获取方式，值得投入

### 3. 多模型路由
**时间**: 2026-03-07  
**结果**: 本地模型节省成本，响应快  
**经验**: 本地+云端混合是最佳方案
